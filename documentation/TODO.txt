
EXPAND ENEMY FUNCTIONALITY:
* I want to achieve:
   - Several hitboxes per enemy 
   - Several animations per enemy 
   - Animations can be rotated
   - LevelEditor can give the enemy a directional vector, which can be used to extract
     rotation, xSpeed and ySpeed.
   - Enemy can become visible/interactable once it reaches a certain point on the screen

* Migration plan:
   - 1. All enemies will have a list of hitboxes, instead of just one.
      .There will be one 'anchor' hitbox, which serves as the main hitbox.
      .For most enemies this will be the only hitbox there is.
      .Include a check in BaseEnemy which enforces that mainHitbox given as argument is not null.
      .LevelEditor will display only this main hitbox. And only this main hitbox will be
       saved to the levelData sheet. The enemy itself will take over the responsibility of
       constructing the remaining hitboxes.
   - 2. Implement rotation for enemies.
        .Every enemy must have a method 'getRotation()'. This can return 0 as default (= no rotation).
        .For now we can have a single rotation for the entire enemy. It's possible we might want
         to have enemies in the future with multiple animated components where each have their own rotation.
         But for now let's keep it to just one common rotation per enemy. Maybe this will be sufficient in the future.
   - 3. Rendering. Make 'drawRotatedImage'-method, and use for all enemies.
        We can look at 'drawRotatedBossPart' for inspiration. We'll probably have to implement a 
        custom 'centerX' and 'centerY' for each enemy, since this will be used as the origin
        for rotation, and some enemies might have origins at different places.
   - 4. Implement support for multiple animated components for each enemy.
       .Idea:
         .Have one sprite sheet per enemy.
         .Have every enemy have a method 'getAniIndexes', which returns a list of tuples: List<(Int, Int)>
         .The idea is - Every entry in the list will represent the row and col in the animation sheet.
         .And each entry in the list will match to a corresponding hitbox.
         .It's up to each enemy to decide how it wants to keep track of these indexes. It might be 
          via AnimatedComponents, SimpleAnimations, or otherwise.
   - 5. Implement the initial code for the Centipede
        .It can just move straight down the screen, for now.
        .We want to verify that the multiple hitboxes are working, and are being rendered properly.
        .We can also rename the onScreen-method to isActive(). Most enemies will be active once they first appear on screen.
         But some enemies (like the Centipede) might only become active once they reach a certain point on the screen.
   - 6 .Implement support for adding vectors to enemies in the LevelEditor.
        .This vector must be added to the levelData somehow.
   - 7. Finish Centipede implementation
        .Extract the vector when constructing the Centipede enemy.
        .The vector can be used to extract both rotation for the enemy, and their movement xSpeed and ySpeed.
   - 8. That should be everything! Hope it works :P
